# 调优任务 2：性能优化（缩短单次运行时间）

> 致执行者：单次 marketnews 运行太慢,根因是**打分与 view 生成在顺序调 DeepSeek**(一次跑约 89 次打分 + ~20 次 view ≈ 100+ 次串行 API 往返)。本任务把这些改成**并发 + 批量**,目标把 15-20 分钟的运行降到几分钟。逐项改、每项跑 `uv run pytest` 确认不破坏测试、单独 commit。改完跑一次 `uv run horizon --marketnews --hours 24` 并报告耗时对比。

## A. 打分并发化（最大且最安全的收益）★

**问题**:`src/marketnews_orchestrator.py` 第 4 步用 `for sid in analyze_story_ids: await analyzer.analyze(...)` 顺序跑。

**改法**:用 `asyncio.gather` + `Semaphore` 并发(默认并发 6,可配),DB 写入仍在 await 之间同步进行(asyncio 单线程,不会并发写 SQLite)。

```python
import asyncio
sem = asyncio.Semaphore(self.config.get("llm_concurrency", 6))
async def _analyze_one(sid):
    primary = self._primary_raw(sid)
    if not primary:
        return
    async with sem:
        res = await analyzer.analyze(primary.title, primary.summary or "")
    if res:
        persister.persist(sid, res)
await asyncio.gather(*[_analyze_one(sid) for sid in analyze_story_ids])
```

- commit: `perf(orchestrator): concurrent scoring with semaphore`

## B. view 生成并发化 ★

**问题**:`src/view/brief_assembler.py` 的 `build` 里 `for rank,(story_id,_score) in enumerate(scored): ... await self.view.generate_item(...)` 顺序跑。

**改法**:并发生成各条 view(保序),再按 rank 写库,最后汇总 market_view。

```python
import asyncio
sem = asyncio.Semaphore(concurrency)  # 同 llm_concurrency
async def _make(rank, story_id):
    primary = self._primary_raw_item(story_id)
    title = primary.title if primary else ""
    raw = primary.summary if primary else ""
    async with sem:
        summary, view = await self.view.generate_item(story_id, title, raw)
    return rank, story_id, title, raw, summary, view
made = await asyncio.gather(*[_make(r, sid) for r,(sid,_s) in enumerate(scored, start=1)])
for rank, story_id, title, raw, summary, view in made:
    item_views.append(view)
    self.items.add(BriefItemRow(brief_id=brief_id, story_id=story_id, rank=rank,
        headline=title, summary=summary, view_text=view, created_at=now))
market_view = await self.view.generate_market_view(item_views)
```

- 注意：DeepSeek 有速率限制,并发别设太高,**6 起步**,若报 429 再降到 3-4。
- commit: `perf(view): concurrent per-item view generation`

## C. 打分批量化（大幅减少调用次数,中等改动）

**问题**:每个 story 单独一次打分调用 → 89 次。

**改法**:`FinancialAnalyzer` 增加 `analyze_batch(items: list[(story_id,title,summary)]) -> dict[story_id, AnalysisResult]`,一次把 ~15 条的标题+摘要塞进一个 prompt,要求模型返回 JSON 数组(每条含 score/tags/entities)。89 次 → ~6 次。

- prompt:在 `financial_prompts.py` 增 `build_batch_score_user(items, taxonomy_codes)`,要求返回 `[{"idx":0,"score":..,"tags":..,"entities":..}, ...]`。
- orchestrator 第 4 步改用 `analyze_batch`(分块,每块 15 条,块间可并发)。
- 解析失败的块只跳过该块、不致命;保留单条 `analyze` 作为兜底。
- 新增测试:假 AI 返回数组 JSON,验证 `analyze_batch` 正确拆分映射。
- commit: `perf(analysis): batch scoring (N stories per LLM call)`

> 收益:A+B 把"串行等待"变并发(约 5-6x);C 再把打分调用数从 ~89 降到 ~6。三者叠加,单次运行有望从 15-20 分钟降到 2-4 分钟。

## D. 限制每期条数（减少 view 调用,即时见效）

`data/config.json` 加/调:
- `min_score`: 6.5（略提阈值,过滤更狠）
- `max_items`: 12（每期最多 12 条 view,而非 20）

view 是最贵的每条调用;12 条足够日读,且省时省 token。`BriefAssembler.build` 已支持这两个参数,确认从 config 读取。

- commit: `perf(config): cap max_items and raise min_score`

## E. 确认运行时模型为 deepseek-v4-flash

确认 `data/config.json` 的 model 是 `deepseek-v4-flash`(非 reasoner/思考模型,也非 pro)——flash 单次响应最快,最适合这种高频调用。若 tuning-1 已改则跳过。
