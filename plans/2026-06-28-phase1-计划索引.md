# Phase 1 实施计划索引（路线图）

> **致执行者（Claude Code + DeepSeek）：** 本系统基于 fork 开源项目 Horizon（github.com/Thysrael/Horizon，MIT）二次开发。设计依据见 `docs/specs/2026-06-28-金融市场brief-设计文档.md`。Phase 1 拆为 7 个子计划，**按顺序执行**，每个子计划产出可独立测试的软件。用 `superpowers:subagent-driven-development` 或 `superpowers:executing-plans` 逐任务执行。

**总目标：** fork 并改造 Horizon，跑通「抓取 → 持久化去重 → 跨天聚类 → 金融打分/分类 → 中性 view 生成 → 入库归档 → 展示」的金融市场每日 Brief 个人版。

**全局约束（每个子计划都隐含适用）：**
- 语言/运行时：Python 3.11+，沿用 Horizon 的 `uv` + `pyproject.toml`。
- 数据库：**SQLite + sqlite-vec**（向量检索扩展）。Phase 1 本地零运维。
- Embedding：**本地免费模型** `sentence-transformers`（默认 `paraphrase-multilingual-MiniLM-L12-v2`，多语言，384 维），零成本、不依赖外部 API。
- LLM：DeepSeek（Horizon 已原生支持，`provider=deepseek`、`model=deepseek-chat`、`DEEPSEEK_API_KEY`）。
- 数据校验：所有跨边界数据用 Pydantic 模型。
- 工程原则：异步并发 I/O、DB 索引 + 向量索引、条件抓取/缓存、批量限流 LLM、TDD、频繁提交。**不做** dynamic typing / JIT kernel / 底层重写。
- 🔒 **合规第一原则**：view 只做中性影响陈述，绝不给投资建议（无买卖方向、无目标价）。每期 brief 带免责声明。输出层做建议性措辞检查。

## Horizon 现状（复用 vs 新建）

**可直接复用：**
| 模块 | 文件 | 作用 |
|---|---|---|
| 统一条目模型 | `src/models.py` → `ContentItem` | 各源归一化的条目（含 ai_score/ai_summary/ai_tags） |
| AI 配置 | `src/models.py` → `AIConfig`、`AI_PROVIDER_DEFAULTS` | 已含 DeepSeek 默认配置 |
| 抓取器框架 | `src/scrapers/base.py` → `BaseScraper.fetch(since)` | 抽象基类，新源继承即可 |
| RSS 抓取器 | `src/scrapers/rss.py` → `RSSScraper` | feedparser 实现，加源=改配置 |
| AI 客户端 | `src/ai/client.py` → `create_ai_client` | 支持 DeepSeek |
| 打分器 | `src/ai/analyzer.py` → `ContentAnalyzer.analyze_batch` | 并发打分 |
| 总结器 | `src/ai/summarizer.py` → `DailySummarizer.generate_summary` | 生成 markdown |
| 补充器 | `src/ai/enricher.py` → `ContentEnricher` | 联网补背景 |
| 提示词 | `src/ai/prompts.py` | 现为科技向，需改金融向 |
| 编排管线 | `src/orchestrator.py` → `HorizonOrchestrator.run` | fetch→merge→score→filter→topic-dedup→enrich→summarize→deliver |
| 投递 | `src/services/{email,webhook}.py`、`docs/`(GitHub Pages) | 邮件/Webhook/静态站 |

**Horizon 没有、需新建（我们的核心改造）：**
1. **持久化关系层**：Horizon 是无状态、文件式（`src/storage/manager.py` 仅存配置+markdown）。我们整套数据模型（raw_item/story/score/enrichment/brief/brief_item/taxonomy/story_tag/entity/story_entity/watchlist + 产品层）全是新建。→ **子计划 1**
2. **跨天/跨次去重**：Horizon 仅单次运行内去重。我们要全局持久化 dedup_key + embedding 跨天聚类。→ **子计划 2、3**
3. **本地 embedding + 向量索引**：→ 子计划 3
4. **金融化提示词 + 分类 + 实体抽取**：→ 子计划 4、5
5. **金融数据源**（SEC EDGAR/Form D、HKEX 新抓取器）：→ 子计划 2
6. **中性合规护栏**：→ 子计划 5
7. **按维度/watchlist 回顾的展示层**：→ 子计划 6

## 子计划清单与依赖

```
子计划 0  Fork 与项目初始化
        └─→ 子计划 1  持久化数据层 (SQLite + sqlite-vec)         ★地基
                    └─→ 子计划 2  摄取层 + 持久化精确去重 + 金融数据源
                              └─→ 子计划 3  本地 embedding + 跨天 story 聚类
                                        └─→ 子计划 4  金融打分 + 分类 + 实体抽取
                                                  └─→ 子计划 5  中性 view 生成(skill路由) + brief 组装
                                                            └─→ 子计划 6  展示/回顾(按维度+watchlist) + 投递
```

### 子计划 0 — Fork 与项目初始化
**产出：** fork 后的可运行项目骨架，DeepSeek 配置就绪，baseline 跑通，pytest 测试框架搭好。
**关键：** `uv sync`；`.env` 填 `DEEPSEEK_API_KEY`；`data/config.json` 设 `provider=deepseek`；跑一次默认源确认管线通；建 `tests/` 与 CI。

### 子计划 1 — 持久化数据层（SQLite + sqlite-vec）★地基
**产出：** 完整 SQLite schema（设计文档第 5 节四层 + taxonomy/entity/watchlist）+ 迁移脚本 + Repository(DAO) 层 + Pydantic 行模型。详见 `2026-06-28-subplan-1-持久化数据层.md`。
**接口产出（后续依赖）：** `Database`、各 Repository（`RawItemRepo`/`StoryRepo`/`ScoreRepo`/`EnrichmentRepo`/`BriefRepo`/`TaxonomyRepo`/`EntityRepo`/`WatchlistRepo`），及 `upsert_raw_item(...)→(id,is_new)` 等签名。

### 子计划 2 — 摄取层 + 持久化精确去重 + 金融数据源
**产出：** 把 Horizon scrapers 的 `ContentItem` 落库为 `raw_item`，按全局 `dedup_key` 幂等 upsert（已存在跳过，零 token）；新增金融 RSS 源配置（Reuters/FT/CNBC/SCMP/Nikkei/财新等）；新增 `SECEdgarScraper`（含 Form D）与 `HKEXScraper`。
**消费：** 子计划1 的 `RawItemRepo.upsert_raw_item`。
**产出接口：** `IngestionService.ingest(since)→IngestStats`；`SECEdgarScraper`/`HKEXScraper`(继承 `BaseScraper`)。

### 子计划 3 — 本地 embedding + 跨天 story 聚类
**产出：** `EmbeddingService`（sentence-transformers，落 `raw_item.embedding`）；sqlite-vec 向量表与 KNN 检索；`StoryClusterer`：新 item 与最近 N 天 story 比相似度，命中则挂为更新（`story.status=updated`）、否则建新 story。
**消费：** 子计划1 的 `StoryRepo`/`StoryMemberRepo`，子计划2 的 raw_item。
**产出接口：** `EmbeddingService.embed(texts)→List[vector]`；`StoryClusterer.assign(raw_item)→(story_id, is_new_story)`。

### 子计划 4 — 金融打分 + 分类 + 实体抽取
**产出：** 改写 `prompts.py` 打分提示词为「金融市场/估值重要性」；扩展 `ContentAnalyzer` 输出分类标签（五轴 taxonomy）与实体；持久化 `score`/`story_tag`/`entity`/`story_entity`。打分仅读标题+摘要（省 token）。
**消费：** 子计划1 Repos，子计划3 的 story。
**产出接口：** `FinancialAnalyzer.analyze(story)→{score, tags, entities}`。

### 子计划 5 — 中性 view 生成（skill 路由）+ brief 组装
**产出：** view 提示词模板库（按设计文档第 8 节路由矩阵，从 morning-note/earnings-analysis/sector-overview 等 skill 提炼，强制中性、无建议）；按 story 分类标签路由模板；生成每条 `view_text` + 整期 `market_view_text`；输出层建议性措辞检查；持久化 `brief`/`brief_item`。
**消费：** 子计划1 Repos，子计划4 的 score/tags。
**产出接口：** `ViewGenerator.generate_item_view(story)→str`；`ViewGenerator.generate_market_view(stories)→str`；`ComplianceChecker.scan(text)→List[Violation]`；`BriefAssembler.build(period)→brief_id`。

### 子计划 6 — 展示/回顾 + 投递
**产出：** 从 DB 渲染每日 brief（Markdown/静态站，复用 Horizon `docs/` + 投递）；按 market_type/行业组/产品组/地区/资产类别筛选的回顾查询；按 watchlist 公司的单实体纵向回顾查询；每期附免责声明。
**消费：** 子计划1 Repos，子计划5 的 brief。
**产出接口：** `BriefRenderer.render(brief_id)→markdown`；`ReviewQuery.by_taxonomy(...)`、`ReviewQuery.by_entity(entity_id)`。
