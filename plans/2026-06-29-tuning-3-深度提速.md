# 调优任务 3：深度提速（批量 embedding + 去二次扫描）

> 致执行者：tuning-2 后单次 ~8 分钟，瓶颈不再是 LLM，而是 ①逐条 embedding ②聚类的全表线性扫描 ③抓取超时过长。本任务做算法层优化（非微优化、非换模型），目标 8 分钟 → 2-4 分钟。每项 TDD、跑分目录测试(`tests/embedding tests/clustering` 等，**勿跑全量以免 OOM/137**)、单独 commit。改完跑一次 `uv run horizon --marketnews --hours 24` 报告耗时。

## A. 批量 embedding（最大收益）★

**问题**：`src/clustering/clusterer.py` 的 `assign()` 对每条 raw_item 调一次 `embedding.embed_one()`，170 条 = 170 次模型推理，单次开销巨大。

**改法**：在编码层批量化。
1. `src/marketnews_orchestrator.py` 第 3 步聚类前，**先把本次所有新 raw_item 的文本一次性编码**：
   ```python
   texts = [f"{r.title}\n{r.summary or ''}" for r in new_raw_ids]
   vecs = EmbeddingService().embed(texts)   # 一次 encode 整个列表
   ```
2. 给 `StoryClusterer` 增加 `assign_with_vec(raw_item, vec)`（或让 `assign` 接受可选预算 `vec` 参数），用传入的向量，不再内部 `embed_one`。
3. orchestrator 改为：
   ```python
   for row, vec in zip(new_raw_ids, vecs):
       sid, is_new = clusterer.assign(row, vec=vec)
   ```
- 测试 `tests/clustering`：用假 embedding 验证 `assign(row, vec=...)` 走传入向量、行为不变。
- commit: `perf(embedding): batch-encode all new items in one call`

## B. 去掉聚类的 O(n²) 全表扫描（第二大收益）★

**问题**：`clusterer.py` 的 `_story_of_item()` 用 `for s in self.stories.recent("0000")` 扫**所有** story 再遍历成员来反查"某 raw_item 属于哪个 story"；`_recent_member_item_ids()` 也线性聚合。每条新 item 都触发，整体接近 O(items × stories)。

**改法**：加反向直查。
1. `src/db/schema.sql`：给 `story_member(raw_item_id)` 加索引
   ```sql
   CREATE INDEX IF NOT EXISTS idx_story_member_raw ON story_member(raw_item_id);
   ```
   并在 `src/db/migrate.py` 的 `ensure_web_columns`（或新增 `ensure_indexes`）里幂等补这条索引（兼容已存在的库）。
2. `StoryMemberRepo` 增加 `story_id_for_raw_item(raw_item_id) -> int|None`（一条 SQL 直查），`StoryClusterer._story_of_item` 改用它，删掉全表扫描。
3. `_recent_member_item_ids`：改为一条 SQL 取「最近 N 天 story 的成员 raw_item_id 集合」（join story + story_member + 时间过滤），不要 Python 循环聚合。
- 测试 `tests/db` + `tests/clustering`：验证反查正确、聚类行为不变。
- commit: `perf(clustering): index-based reverse lookup, drop O(n^2) scans`

## C. 降低抓取超时 + 并发抓取（不限每源条数）

**问题**：`marketnews_orchestrator.py` 的 `httpx.AsyncClient(timeout=30.0)`，失效源干等 30s。

**改法**：
- 超时降到 `timeout=10.0`（绝大多数 RSS 2 秒内返回，10s 足够；只砍卡死/失效源）。
- 确认抓取是并发的（Horizon 的 `fetch_all_sources` 本就并发；本项目 orchestrator 若是 `for sc in scrapers: await sc.fetch()` 顺序，则改成 `asyncio.gather` 并发抓取）。
- ⚠️ **不要新增「每源限条数」上限** —— 保持各源原有返回条数，优先保覆盖面（去重 + 打分阈值已能控量）。
- commit: `perf(fetch): lower timeout to 10s, concurrent fetch`

## D.（可选）Apple GPU 加速 embedding

收益有限（模型加载占大头、我们已批量化），但 Apple Silicon 本地跑可顺手开：
- `EmbeddingService` 初始化模型时尝试 `device="mps"`，失败回退 CPU：
  ```python
  import torch
  device = "mps" if torch.backends.mps.is_available() else "cpu"
  SentenceTransformer(self.model_name, device=device)
  ```
- 仅本地有感；GitHub Actions(无 GPU)自动回退 CPU，不影响。
- commit: `perf(embedding): use MPS device when available`

## 收尾
- 分目录跑测试确认无回归（**勿一次跑全量**，embedding 测试吃内存易 137）。
- `uv run horizon --marketnews --hours 24` 报告新耗时（目标 2-4 分钟）、各阶段大致占比。

## 不做（ROI 低）
- 不重写"热点 Python 代码"/不上 Cython/JIT —— 本系统瓶颈是网络+模型推理，微优化无意义。
- 不换 embedding 模型 —— 批量化后收益已足够；换模型会动维度、影响向量表。
