# 金融市场每日 Brief 系统 — 设计文档

> 版本：v1.0 ｜ 日期：2026-06-28
> 状态：设计已确认，待转入实施计划
> 用途：本文档供 Claude Code + DeepSeek 作为实现蓝本

---

## 1. 项目概述

构建一个**定时（每日 / 每 12 小时）运行**的系统，自动抓取市面上有关 **M&A、投融资、一级市场（PE/VC）与二级金融市场**的信息，经清洗、去重、分类、分析后，生成一份**结构化的金融市场 Brief**。

每条信息保留**原文链接**供深入阅读；系统对每条信息给出**中性的影响陈述（view）**，并在每期末尾给出**综合市场观点**。

**核心目标**：
1. 帮助使用者高效掌握每日金融市场与一级市场动态。
2. 培养 market sense（使用者为 PE 实习生，目标为投行 IBD / Equity Research）。
3. 长期做成**可发布的产品**，让他人使用，并采集使用数据与反馈以迭代改进。

---

## 2. 范围与分阶段

系统按两阶段推进，但**数据模型一次性设计到位**，避免将来推倒重来。

### Phase 1（先做）— 个人版
fork 并改造 Horizon，跑通完整管线：抓取 → 去重 → 打分过滤 → 分类/验证 → 生成 view → 入库归档 → 展示（静态站 / 邮件）。先供使用者自用、迭代 prompt 与数据源。使用一个固定的「默认用户」，产品层表结构建好但不激活。

### Phase 2（之后做）— 产品版
在同一套数据之上叠加产品层：用户账户、托管服务、行为埋点（使用数据）、反馈收集。坚持**「中央统一生成、用户只筛选不重算」**的架构（见第 7 节），保证 API 成本与用户数无关。

---

## 3. 技术路线：fork 改造 Horizon

基于开源项目 **Horizon**（github.com/Thysrael/Horizon，MIT 许可，Python）二次开发。

**选择理由**：Horizon 的七步管线（Fetch → Deduplicate → Score & Filter → Enrich → Summarize → Deliver）与本系统需求高度一致；原生支持 **DeepSeek** 及任意 OpenAI 兼容 API；内置金融新闻源（OpenBB）、RSS 源、GitHub Pages 每日站点发布、邮件 / Webhook 投递、GitHub Actions 定时运行；最影响稳定性的「管道工程」已被社区验证。MIT 许可允许商用（保留许可声明即可）。

**改造重点**（不动核心骨架）：
1. **换数据源**：替换为金融与一级市场源（见第 4 节）。
2. **改 prompt**：打分提示词改为「对市场 / 估值的重要性」；总结提示词改为输出**中性影响 view**（见第 8 节）。
3. **接 DeepSeek**：配置中将模型指向 DeepSeek API。
4. **扩数据模型**：在 Horizon 基础上落地第 5 节的数据模型（分类、实体、watchlist、产品层）。

## 4. 数据源

策略：**摘要 + 链接靠 RSS，硬核一手事件靠官方源**。优先权威来源。

**覆盖原则：全球化、多区域，不固定聚焦单一地区；对亚太 / 大中华 / 香港加重**（对齐使用者求职目标——八大投行及头部 PE 的香港 office，业务为全球 + 亚太跨境）。

| 类别 | 来源举例 | 说明 |
|---|---|---|
| 全球 / 英文主流 RSS | Reuters、Bloomberg RSS、FT、WSJ、CNBC、MarketWatch | 标题 + 摘要 + 原文链接；正文留链接给用户点进读 |
| 亚太 / 大中华（重点） | Nikkei Asia、SCMP（南华早报）、财新、财联社、华尔街见闻 | 加重亚太与大中华区覆盖 |
| 官方一手源 · 美国 | SEC EDGAR（8-K 重大事件 / 并购、13D/G、S-1）+ **Form D**（私募融资 / 私募配售申报）、美联储发布 | 权威、免费；Form D 是一级市场难得的免费硬数据源 |
| 官方一手源 · 香港 | **HKEX「披露易」公告**（香港上市公司公告，等价于美国 EDGAR） | 香港权威一手源，针对 HK office 求职极有价值 |
| 官方一手源 · 内地 | 巨潮资讯（A 股公告）、交易所公告、央行发布 | 大中华区一手 |
| 一级市场行业媒体 | PE Hub、Axios Pro Rata、TechCrunch、FinSMEs 等 RSS | 融资 / 并购 / 募资动态 + 原文链接 |
| 可选付费（Phase 2） | NewsAPI / Finnhub / Marketaux（二级）；PitchBook / Crunchbase / 投中 / 清科（一级） | 覆盖最全、最稳定，但有费用 |

**现实预期**：Phase 1 免费源（Form D + HKEX 公告 + 多区域行业媒体 RSS）能覆盖相当一部分全球及一级市场动态；机构级全覆盖需将来接付费数据源。

> 注意：不直接抓取付费墙后的正文（Bloomberg/WSJ/FT 正文），仅取其免费 RSS 摘要并保留原文链接，规避反爬与合规风险。

## 5. 数据模型（核心）

数据模型分四层。**三条贯穿全局的稳定性原则**：

1. **原始层不可变、只追加**：抓回的新闻入库后不再修改；所有加工均为「派生数据」，可删除重算，原始真相永不被污染。
2. **幂等去重**：每条新闻有稳定指纹（规范化 URL + 内容哈希）；重复抓取 / 补跑不产生重复记录。
3. **全程可溯源**：最终 brief 的每条都能回溯到依据的原始新闻与原文链接——同时满足「点原文阅读」与「信息验证审计链」。

### 第一层 · 摄取层（Ingestion，不可变）

```
source        —— 数据源注册表
  id, name, type(rss/edgar/api), url, trust_tier(权威分级),
  language, category, enabled, fetch_config(JSON), created_at

raw_item      —— 每条抓回的原始新闻（只追加，不修改）
  id, source_id(→source), external_id(源自带 guid),
  url, canonical_url, title, summary, author,
  published_at, fetched_at,
  dedup_key(规范URL+内容哈希，唯一索引，★全局持久化、跨天有效), content_hash,
  embedding(向量, 用于跨天语义相似度比对), 
  language, raw_payload(JSON, 存原始全量以备重解析)
```

### 第二层 · 加工层（Processing，可重算）

```
story         —— 把讲同一件事的多条新闻合并成「一个故事」（跨天持续存在）
  id, canonical_title, dedup_cluster_key,
  first_seen_at, last_seen_at, last_update_at,
  status(新增 new / 进行中 ongoing / 有更新 updated / 沉寂 dormant)

story_member  —— story 与 raw_item 的多对多关联（溯源核心）
  story_id(→story), raw_item_id(→raw_item), is_primary

score         —— AI 打分（省 token 关卡：低分不进入后续昂贵步骤）
  id, story_id(→story), model, score(0-10),
  importance_rationale, scored_at
  （存历史：换 prompt 重打分，旧分仍可对比）

enrichment    —— 重要 story 的背景补充 + 验证信息
  id, story_id(→story), context_text,
  corroborating_sources(JSON, 交叉验证的其他源),
  confidence(已证实 / 待证实), created_at
```

### 第三层 · 输出层（Output，即 Brief）

```
brief         —— 一期 brief（每日 / 每 12 小时一期）
  id, period_type(daily/12h), period_date,
  language, model, generated_at, status,
  market_view_text(★整期末尾的综合市场观点)

brief_item    —— brief 里的一条（含单条 view）
  id, brief_id(→brief), story_id(→story),
  rank, headline, summary,
  view_text(★该条新闻对市场/估值的中性影响陈述),
  created_at
  （行业/产品/地区/资产类别等分类从所属 story 继承）
```

### 分类、实体与 watchlist

```
taxonomy   —— 受控词表（固定分类，保证回顾时筛得干净），五个维度：
  dimension ∈ { market_type, industry_group, product_group, region, asset_class }
  id, dimension, code, label, sort_order

  · market_type    : 一级市场/primary, 二级市场/secondary
  · industry_group : TMT, Consumer & Retail, Healthcare, Energy & Natural Resources,
                     Industrials, FIG(金融机构组), Real Estate ...
  · product_group  : M&A/Advisory, ECM, DCM, LevFin, Restructuring,
                     VC融资(分轮次), PE Buyout, 基金募集/Fund Closing,
                     Pre-IPO, 私募配售/Private Placement
  · region         : 北美, 欧洲, 大中华(含香港), 日韩, 东南亚·印度, 新兴市场, 全球·跨区域
  · asset_class    : 股票, 债券, 私募股权, 外汇, 大宗商品 ...

story_tag  —— story 与 taxonomy 的多对多（一条 story 可同时挂多个轴 / 多个标签）
  story_id(→story), taxonomy_id(→taxonomy)
  （例：某科技公司被收购 = TMT + M&A + 大中华 + 股票）

entity     —— 新闻涉及的公司 / 标的（支持未上市公司）
  id, type(company/ticker/sector/person),
  name, ticker, identifiers(JSON: ISIN/CIK 等), created_at

story_entity —— story 提到了哪些实体（溯源 + 单公司回顾核心）
  story_id(→story), entity_id(→entity), role(主角/相关方)

watchlist_item —— 关注的公司清单（Phase1 用默认用户，Phase2 接真实用户）
  id, user_id, entity_id(→entity), note, added_at
```

### 第四层 · 产品层（Product，Phase 2 实现，现在预留结构）

```
user          —— id, email, name, plan, created_at
subscription  —— user_id, 订阅的 source/topic, 阈值, 推送渠道, 语言偏好
delivery      —— user_id, brief_id, channel, sent_at, status
event         —— user_id, brief_item_id, type(open/click/dwell), ts   ★使用数据埋点
feedback      —— user_id, brief_item_id, rating, comment, ts          ★体验反馈
```

**回顾能力**：基于上述结构，使用者可按 **一级/二级、行业组、产品组、地区、资产类别** 任意筛选历史 brief；点开某个 watchlist 公司，可调出**所有提到它的 story 与历史判断**，做单公司纵向回顾（含未上市公司）。

## 6. 系统流程（七步管线 · 金融场景落地）

```
1. Fetch 抓取   —— 并发拉取金融源(权威RSS + SEC EDGAR + Form D + 可选API)，按 24h/12h 时间窗
2. Dedup 去重   —— ★两段式（均在昂贵模型调用之前，故省 token）：
                  (A) 精确去重：全局持久化 dedup_key 查重，已存在直接丢弃（跨天有效，零 token）
                  (B) 跨天语义聚类：用 embedding 与最近 N 天的 story 比相似度，
                      命中则挂为该 story 的"更新"，仅实质进展时生成增量 view，否则跳过
3. Score 打分   —— ★Token 关卡：只读「标题+摘要」打 0-10 分，留高分项，低分丢弃
4. Enrich 加工  —— 对高分项：跨源交叉验证(标记 已证实/待证实) + 抽取实体(公司)
                  + 按五轴分类打标签(market_type/行业组/产品组/地区/资产类别)
5. View 生成    —— ★调用 DeepSeek：先按第 4 步的分类标签路由到对应 view 提示词模板
                  (见第 8 节路由矩阵)，再生成每条中性影响陈述 + 整期末尾综合市场观点；
                  模板内嵌从对应 skill 提炼的分析框架；强制中性、无建议
6. Store 入库   —— brief / item / 标签 / 实体关系写入数据库（可回顾归档）
7. Serve 服务   —— Phase1: 静态站/GitHub Pages + 邮件；
                  Phase2: 只读 API 给用户端按订阅/watchlist 筛选
```

第 3、4、5 步将昂贵的模型调用集中在「少数高分项」上，且分类 / 验证尽量并入同一次调用——这是省 token 的落地方式。

---

## 7. Token 经济学（双层省 token）

**第一层 — 生成时过滤**：先用廉价方式（去重、规则 / embedding 聚类）浓缩，AI 打分只读「标题+摘要」，**仅高分项进入昂贵的 Enrich + View 生成**。分类标签在 View 生成的同一次调用里顺带输出，几乎零额外成本。

**跨天去重（额外省 token）**：dedup_key 全局持久化，完全重复的新闻在入库前即被丢弃（零 token）；同一事件的后续报道经 embedding 比对挂到已有 story，仅实质进展才生成增量 view。即"老调重弹"被挡在昂贵步骤之前——跨天去重是省 token 而非耗 token。

**第二层 — 服务时只读（产品级核心）**：所有花 token 的环节在**中央、每期只跑一次**，结果连同分类一并入库。用户端只做**读取与按分类筛选**，不触发任何模型调用。
→ API 成本为 **O(信息量)**，而非 O(信息量 × 用户数)。1 个用户与 1 万个用户，token 账单几乎一致。

**设计约束（务必遵守）**：中央管线生成**一套统一的、已分类的** brief 语料；用户**订阅 / 筛选**自己关心的组（如只看 TMT + M&A），**不允许**演变成「每个用户自定义源 → 各自跑模型」，否则成本将随用户数爆炸。

## 8. View 生成（中性合规 + 借鉴 skill 框架）

### 🔒 合规第一原则（不可违背）
产品**只做「信息汇总 + 中性影响陈述」，绝不提供投资建议**。view 一律：
- 中性陈述影响，**不给买卖方向、不给目标价**；
- 不使用「建议 / 看多 / 看空 / 应该买入」等措辞；
- 每期 brief 附免责声明（「信息汇总，非投资建议」）；
- 输出层加一道**建议性措辞的轻量检查**，拦截违规表述。

### 借鉴 Anthropic 金融 skills 的分析框架
Skill 依赖 Claude agent 环境运行，**DeepSeek 无法直接加载**。因此采用与 fork Horizon 相同的策略——**借逻辑，不依赖运行**：提炼相关 skill 的分析方法论，**烧进 view 生成提示词**，让 DeepSeek 按框架输出。

基础格式参考 **morning-note**（紧凑、结构化的新闻点评格式），但**去掉方向性观点**，改为中性影响陈述。

### Skill 路由矩阵（新闻类型 → 分析框架）
第 4 步已给每条 story 打了分类标签（market_type / industry_group / product_group 等），**直接用这些标签路由到对应的 view 提示词模板**，无需额外判断。每个模板内嵌从对应 skill 提炼的分析框架。

**🔬 Equity Research 视角（个股 / 估值类新闻）**

| 新闻类型 | 借鉴 skill 框架 | 提炼什么 |
|---|---|---|
| 财报发布（已出） | `earnings-analysis` | beat/miss、盈利质量、guidance 诊断结构 |
| 财报前瞻 | `earnings-preview` | 关注指标、预期框架 |
| 个股首次覆盖 / 重大基本面 | `initiating-coverage`、`growth-stock-analysis` | 基本面拆解、护城河 / 增长分析框架 |
| 估值相关 | `comps-analysis`、`dcf-model` | 可比公司倍数、现金流视角（仅作影响陈述，不输出目标价） |

**🏦 IBD / Deals 视角（M&A、融资类新闻）**

| 新闻类型 | 借鉴 skill 框架 | 提炼什么 |
|---|---|---|
| M&A / 并购 | `competitive-analysis`、`comps-analysis`、`corporate-network-analysis` | 战略契合、市场格局、交易倍数、关联方网络 |
| 公司概况速写 | `fsi-strip-profile` | 标的公司一页式画像结构 |
| 杠杆收购 / LBO 融资 | `lbo-model` | 杠杆结构、回报来源（仅陈述机制） |
| 卖方流程类 | `teaser`、`process-letter` | 交易进程解读 |

**📊 Asset Management / 宏观主题视角**

| 新闻类型 | 借鉴 skill 框架 | 提炼什么 |
|---|---|---|
| 行业 / 板块动态 | `sector-overview` | 行业动态、竞争格局框架 |
| 宏观 / 主题趋势 | `thematic-investment-research`、`supply-chain-pass-through` | 主题验证、产业链传导分析 |
| 未来事件 | `catalyst-calendar` | 把新闻抽成催化剂时间线 |
| 持续跟踪 | `thesis-tracker` | 对 watchlist 标的的纵向跟踪框架 |

**🚀 一级市场（PE/VC）视角**

| 新闻类型 | 借鉴 skill 框架 | 提炼什么 |
|---|---|---|
| 创投融资 / 新兴赛道 | `embodied-ai-sourcing`、`growth-stock-analysis`、`thematic-investment-research` | 赛道评估、商业化路径、主题框架 |

### 两个不可违背的原则
1. **⚠️ 合规过滤（凌驾于所有 skill 之上）**：`initiating-coverage`、`idea-generation`、`growth-stock-analysis` 等 skill 本身带**方向性结论**（看多 / 看空 / 目标价 / pitch）。我们**只取其分析骨架（该看哪些维度），剥掉一切推荐性输出**，最终仍是中性影响陈述。
2. **两种运行模式**：
   - **日常 brief（便宜，DeepSeek）**：将上述框架**提炼成提示词模板**，按分类路由选用——**不真正运行 skill**。
   - **深度分析（Phase 2 付费，Claude agent）**：用户对某 watchlist 标的点「深挖」时，**才**在 Claude agent 环境中真正运行重型 skill（`dcf-model`、`comps-analysis`、`lbo-model` 等）。

> 实现阶段动作：写代码框架时，实际读取上述各 skill，将其分析框架提炼为按类型分组的、可复用的 view 提示词模板库。

## 9. 错误处理与稳定性机制

- **单源隔离**：某源超时或改版导致抓取失败，仅记录并跳过，不拖垮整次运行。
- **幂等写入**：靠 dedup_key 做 upsert，定时任务重复跑 / 补跑均安全。
- **原始不可变 + 派生可重算**：换 prompt 重新打分 / 重生成 view，删派生层重跑即可，原始数据零损失。
- **LLM 调用韧性**：超时、重试退避、输出按 schema 校验、单条失败只标记不致命、可配置备用模型。
- **成本护栏**：每次运行设上限（最大条目数、分数阈值、token 预算上限），防止新闻爆量烧爆账单。
- **运行监控**：每次运行写状态记录，失败用 Webhook 告警（Horizon 自带）。
- **合规护栏**：view 提示词强制中性 + 输出层检查建议性措辞 + 每期免责声明。

### 性能与工程优化原则

本系统为 **I/O 与 API 密集型**（瓶颈在网络抓取与 LLM API，而非 Python 执行速度）。因此**不采用** dynamic typing 调整、Numba/Cython JIT kernel、底层重写等针对 CPU 密集型数值计算的手段——它们在此属过早优化，徒增复杂度、收益接近零。真正的效率与稳定来自以下工程约束（按收益排序）：

1. **异步并发 I/O**：用 `asyncio` / `aiohttp` 并发抓取所有数据源、批量发起 LLM 请求（最大运行时收益）。
2. **数据库索引 + 向量索引**：`dedup_key` 建唯一索引；embedding 用专门向量索引（FAISS / pgvector / sqlite-vec），不用 Python 循环比对。这是「数据增长后仍稳定」的关键。
3. **缓存与条件抓取**：RSS 用 ETag / Last-Modified 条件请求（无更新则不重传）；纯函数用 `@lru_cache`。
4. **批量 + 限流 LLM 调用**：批处理 + token 预算上限。
5. **Pydantic 数据校验**：在数据进出边界处校验 schema，挡住脏数据（用静态类型提升稳定性，而非提升速度）。
6. **类型注解（type hints）**：用于可维护性与辅助 AI 正确生成代码；CPython 运行时不依赖它提速。

**原则**：先把功能写对、能跑，再用实测数据定位真正热点后才优化；无性能数据前不做底层优化。
